# Plan: Reorganización Docker Compose para Desarrollo

## Objetivo
Centralizar y unificar la configuración Docker en una carpeta `docker/` separada, manteniendo un único docker-compose de desarrollo funcional con red automática, conservando Redis para uso futuro.

---

## Análisis Estado Actual

### Archivos Docker existentes
- `docker-compose.yaml` (root) → usa `include` para backend y frontend
- `backend/docker-compose.yaml` → define: mongodb, api, cache (redis)
- `frontend/docker-compose.yaml` → define: frontend (vite)
- `backend/api/Dockerfile` → multi-stage (deps + dev)
- `frontend/Dockerfile` → multi-stage (deps + dev)

### Problemas detectados
1. **Configuración fragmentada**: 3 archivos compose con `include` (complejidad innecesaria)
2. **Red externa inexistente**: `external: true` en backend compose causa error
3. **Docker Compose unificado deseado**: Un solo archivo más maintainable
4. **Makefile eliminado**: Ya no existe (confirmado en git status)

---

## Propuesta de Solución

### Estructura final deseada
```
shop_books/
├── docker/                      # Nueva carpeta devops
│   ├── docker-compose.dev.yml   # Unificado: backend + frontend + db + cache
│   └── .env.example             # Plantilla variables de entorno
├── backend/
│   └── api/
│       ├── Dockerfile           # (sin cambios)
│       └── .env                 # (ignorado en git)
├── frontend/
│   ├── Dockerfile               # (sin cambios)
│   └── .env                     # (ignorado en git)
├── .gitignore                   # (ya ignora .env)
└── README.md                   # (actualizar sección docker)
```

### Configuración `docker/docker-compose.dev.yml`

**Servicios a incluir:**
1. `mongodb` → mongo:latest, puerto 27017, volumen persistente
2. `api` (backend) → build desde `backend/api`, target `dev`, puerto 3000
3. `cache` (redis) → redis:7.4-alpine, puerto 6379, password admin1234
4. `frontend` → build desde `frontend`, target `dev`, puerto **8080:5173**

**Red:**
- Red definida en el compose (no external)
- Todos los servicios en misma red
- Frontend se comunica con api en `http://api:3000`

**Volúmenes:**
- mongodb: datos persistentes
- api: montar código fuente (hot-reload)
- frontend: montar código fuente (hot-reload)
- node_modules separados (evitar sobrescritura)

**Environment:**
- api: `env_file: ./backend/api/.env`
- frontend: variables de entorno inline o archivo

---

## Pasos de Implementación

1. **Crear carpeta `docker/`** en root
2. **Crear `docker/docker-compose.dev.yml`** completo con:
   - 4 servicios (mongodb, api, cache, frontend)
   - Red automática (sin `external: true`)
   - Volúmenes apropiados para desarrollo
   - Mapeo de puertos: api 3000:3000, frontend 8080:5173, mongodb 27017:27017, redis 6379:6379
3. **Eliminar** `backend/docker-compose.yaml` y `frontend/docker-compose.yaml`
4. **Eliminar** `docker-compose.yaml` del root (que usaba include)
5. **Crear** `docker/.env.example` con variables necesarias (MONGO_ROOT_USERNAME, MONGO_ROOT_PASSWORD)
6. **Actualizar README.md**:
   - Actualizar comando de inicio: `docker-compose -f docker/docker-compose.dev.yml up`
   - Aclarar puertos: frontend http://localhost:8080, api http://localhost:3000
7. **Verificar** que `.env` ya está en `.gitignore` (confirmado)
8. **(Opcional)** Crear alias o script de inicio si se desea

---

## Decisiones tomadas

- **Un solo compose de desarrollo**: Más simple y fácil de mantener
- **Red automática**: Se crea automáticamente con el compose (no external)
- **Mantener Redis**: Para futura implementación (caché, sesiones)
- **Puerto frontend 8080**: Puerto memorable, mapeado a Vite (5173)
- **Variables sensibles**: `.env` ya en `.gitignore`, se crearán localmente
- **Sin Kubernetes**: Se mantiene Docker Compose como solicitado
- **Carpeta `docker/` separada**: Buena práctica DevOps, limpia el root

---

## Notas adicionales

- **Dockerfiles**: No requieren cambios (ya tienen stage `dev` correcto)
- **Hot-reload**: Los volúmenes de código fuente están montados en ambos servicios
- **Dependencias**: `api` depende de `mongodb` (healthcheck). Frontend depende de `api` (opcional, pero recomendado añadir `depends_on` si se necesita orden de inicio)
- **Variable FRONTEND_URL**: En `backend/api/.env` apunta a `http://localhost:8080`, que coincidirá con el frontend accesible
